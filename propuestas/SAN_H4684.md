# #SAN #H4684 Gestión de incidencias de producto y transporte desde webapp

Los objetivos de este proyecto son:

+ Proporcionar a los agentes comerciales acceso para ver y gestionar incidencias de transporte y de producto desde la app web

Jesús

## Estructura

### Webapp / Consulta de incidencias
`DUDA 1: ¿ampliar actual opción de incidencias o usar una nueva https://webapp.sanhigia.com/ss/incidencias ?`

Crearemos una opción Incidencias en la webapp que mostrará a cada agente las incidencias de sus clientes asociados que sean de tipo Transporte o Producto. 

Se definen las incidencias de transporte o producto como aquellas cuya categoría tiene un valor del campo _Tipo causante_ igual a _Transportista_ o _Proveedor_, respectivamente.

Los campos de la lista de incidencias a mostrar son:
    + Fecha
    + Cliente
    + Tipo de incidencia (transporte / producto)

### Webapp / Creación de incidencias
Los agentes podrán crear incidencias de tipo Transporte o Producto para sus clientes.

`DUDA 2: ¿ampliar la asociación de incidencias a más de una referencia. Hay que tener en cuenta que si hay más de una referencia asociada a la incidencia, el proveedor puede ser distinto en cada una de ellas. Quizá es mejor crear dos o más incidencias en estos casos y que tengan un seguimiento independiente. Si se decide asociar más de una referencia a la incidencia, indicar qué hacer si las referencias tienen distinto proveedor por defecto`

Al crear una incidencia el proceso será:

+ Indicar la descripción.
+ Indicar el cliente
+ Indicar el tipo

    + Si el tipo es producto, indicar la factura y elegir `el/los` producto`s`
        + El sistema asocia a la incidencia el proveedor por defecto del producto indicado (ver `DUDA 2`)

    + Si el tipo es transporte, indicar la factura
        + El sistema asocia a la incidencia el transportista asociado al albarán de la factura indicada.

+ Indicar obligatoriamente las observaciones


### Artículos e incidencias / Nuevos campos

+ Artículos
    + Meses de garantía. Valor entero. Nulo o cero indican que el producto no tiene garantía.

+ Incidencias
    + En Garantía. `Suponemos que DUDA2 = 1 referencia por incidencia`. Será Sí cuando la fecha de la incidencia es menor a la fecha de la factura + número de meses de garantía.

+ Configuración de CRM
    + Correos / Usuarios de notificación de nueva incidencia, separados por ";".

### Incidencias / Notas
El usuario podrá crear notas asociadas a la incidenca. Los campos mínimos de estas notas serán:

+ Descripción
+ Fecha (automático)
+ Usuario / Agente (automático)
+ Incidencia (automático)

### Incidencias / Informe hoja de cálculo
Crearemos una opción de informe de incidencias.

Criterios de búsqueda:

    + Agente asociado
    + Fecha desde - hasta de creación

Columnas a mostrar:

    + Fecha
    + Tipo (Transporte o producto)
    + Cliente (código + nombre)
    + Artículo`/s` (referencia + descripción)
    + Estado
    + Observaciones
    + Notas (concatenadas con intros indicando fecha + descripción)
    + Proveedor / Transportista (código + nombre), según el tipo
    + Código de incidencia

### Incidencias / Notificación
Al crearse una nueva incidencia se enviará un correo o notificación a la lista de destinatarios de notificación de nueva incidencia.

## Texto original 
Incidencias
Gestión de incidencias. 
Queremos incorporar las siguientes funcionalidades a las incidencias de tipo incidencia de producto o incidencia de transporte.
    1. Queremos que cada comercial vea incidencias asociadas a clientes asignados a él. 
    2. También que las pueda generar.
    3. Cuando genere una incidencia de esta tipología, debe elegir primero el cliente asociado a la incidencia. Caso de ser incidencia de producto, elegir la factura y el producto asociado a la incidencia, pueden ser 1 o más productos. El proveedor se informa en las incidencias de producto automáticamente con el artículo escogido y en las de transporte se elige la factura en incidencia, para así informar automáticamente del proveedor de transporte.
    4. En las incidencias de producto, debemos saber si el producto está en garantía o no, para ello, debemos incorporar un check en artículos que sea si el producto tiene garantía o no, y caso de tenerla una casilla con el periodo de garantía en meses.
    5. El campo de si está el artículo en garantía o no en la incidencia sería informado calculando el periodo de garantía, desde la emisión de la factura. Si hay más de un artículo, se escoge el artículo con el periodo de garantía mayor para decidir si está o no.
    6. El campo observaciones debe ser obligatorio, así como el cliente y al artículo caso de ser incidencia de producto.
    7. Queremos incorporar la función de subir documentos, ya sea vídeos, fotos, o pdfs, que se podrán descargar desde el crm.
    8. Se debe incorporar un campo de notas en las incidencias, para informar del estado de las mismas, similar al de los tratos.
    9. Queremos un informe en Excel de incidencias, donde deben figurar los siguientes datos:
        a. Fecha
        b. Tipo
        c. Cliente
        d. Artículo o artículos  (si hay mas de uno, se ponen todos en una línea)
        e. Estado
        f. Observaciones
        g. Notas
        h. Proveedor caso de ser de producto
        i. Transportista caso de ser de transporte
        j. Código de incidencia (puede ser el código autoasignado)
    10. Cuando se abra la incidencia, se debe enviar un correo electrónico a un destino a configurar, o destinos separados por ;.
